# Skannerz Trading Protocol

Reverse-engineered from `trade.csv` — a 200 kHz logic-analyzer capture of a
complete monster trade between two Radica **Skannerz Commander** (model
72045) devices. The Commander has no tribes (unlike the original Skannerz);
its 126 regular monsters are divided into three categories (Tech/Power/Magic
class symbols), plus 12 secret monsters.

Traded monsters:

| Device | Role in capture | Monster | HP |
|--------|-----------------|---------|----|
| 1 | **master** (drives clock) | Night Lurk — wire number byte 0x01 | 3 |
| 2 | **slave** | Diamond Back — wire number byte 0x11 | 1 |

Confidence notes: everything below is read directly from the capture unless
marked *(hypothesis)*. Payload field semantics (§3.5) were additionally
validated by hardware testing with an Arduino slave emulator against a real
toy (2026-07-16). See `IMPLEMENTATION.md` for an MCU bit-bang guide and
`analysis/` for the scripts used.

## 1. Physical layer

3-pin link connector:

| PIN | Function after handshake |
|-----|--------------------------|
| 1 | Clock (capture channels 1 & 2) |
| 2 | GND |
| 3 | Data (capture channels 0 & 3) |

Both signal lines idle **high**. Both devices can drive the data line
(half-duplex, see §2), so treat it as open-drain-style: drive low actively,
release to idle high. Logic level: **4.5 V** — the device runs on three
AAA cells (lower as the batteries drain).

**Clock**: generated by one device only (the *master*), 1408 Hz
(710 µs period, 50 % duty, = 142 samples at 200 kHz). SPI-style summary:
CPOL=1, CPHA=0 — master data is stable at the falling edge.

## 2. Bit transport — one wire, two directions

Both devices transmit on the same data wire during the same clock burst,
divided by clock phase:

- **Master** updates data ~5 µs after the clock **rising** edge.
  Master bits are sampled at the **falling** edge.
- **Slave** updates data ~35 µs after the clock **falling** edge.
  Slave bits are sampled at the **rising** edge.

```
CLK   ────╮       ╭───────╮
          ╰───────╯       ╰────
       ^  ^slave      ^master
       |  updates     updates
       sample master  |
       bit here       sample slave bit at rising edge
```

Each clock cycle therefore carries one *master bit slot* and one *slave bit
slot*. A device that is not transmitting leaves the line alone, so the idle
side's slot passively mirrors the active side's level — receivers must frame
on start bits (below), and a sniffer must attribute traffic by which clock
phase the data-line *transitions* occur in.

The two devices never transmit payload simultaneously; they take turns.
The slot mechanism is what lets the slave reply on the very next cycle
without bus turnaround.

## 3. Frame formats

All bytes are sent **MSB first**. Data line idles high; a start bit is a 0
in the sender's slot.

### 3.1 Master frame (12 cycles)

```
cycle:   0     1..8      9    10    11
        start  data     stop  low   high
         (0)  b7..b0    (1)   (0)   (1)
```

After the stop bit the master always pulls the line low for exactly one
cycle, then releases. This trailer appears even with no partner present
(handshake beacons), so it is part of the master frame format
*(purpose unknown — possibly an end-of-frame / turn-handoff marker)*.

### 3.2 Slave ack frame (10 cycles)

```
cycle:   0     1..8      9
        start  data     stop
         (0)  b7..b0    (1)
```

Sent in the slave slot (bits change just after falling edges). The reply to
a master frame starts exactly **12 cycles after the master frame's start
bit**, i.e. on the cycle right after the master trailer ends.

### 3.3 Event frames — inverted stop slot

Both sides mark certain frames as "events" by sending the stop slot **low**:

**Slave event frame** (byte 0x27 only, 11 cycles):

```
cycle:   0     1..8      9     10
        start  data     low   low   → release high
         (0)  b7..b0    (0)   (0)
```

**Master event frame** (byte 0x39 only, 12 cycles) — the normal tail `1,0,1`
becomes `0,1,1`:

```
cycle:   0     1..8      9     10    11
        start  data     low   high  high
         (0)  b7..b0    (0)   (1)   (1)
```

The slave's reply to a master event frame starts at **start + 13** cycles
(one later than for normal frames).

### 3.4 Master ack pulse (2 cycles)

After every completed slave frame the master pulls the data line low for
~2 cycles and releases. The pull happens at an odd phase, ~520–600 µs after
the falling edge of the slave's final cycle (i.e. ~73–85 % into that cycle),
release just after a rising edge. It can begin during the slave's stop cycle,
so a sniffer sampling slave slots may see the stop bit clobbered.

### 3.5 Monster payload frame (52 cycles)

Sent in the sender's own slot (master timing for master, slave timing for
slave). One long frame, no per-byte start/stop bits:

```
bit    0     1-3    4-11    12-15   16-23    24-31    32-43     44-50   51
      start  sync   NUM     check   HP(BCD)  HP(BCD)  zeros     EXP     stop
       (0)  '111'  (8 bit) (4 bit) (8 bit)  (8 bit)  (12 bit)  (7 bit) (1)
```

Observed payloads:

| | wire bits (52) | NUM | chk | HP | HP | zeros | EXP | stop |
|---|---|---|---|---|---|---|---|---|
| master (Night Lurk, HP 3) | `0111 00000001 0001 00000011 00000011 000000000000 0000000 1` | 0x01 | 1 | 3 | 3 | 0 | 0 | 1 |
| slave (Diamond Back, HP 1) | `0111 00010001 0010 00000001 00000001 000000000000 0000110 1` | 0x11 | 2 | 1 | 1 | 0 | 6 | 1 |

Field semantics (capture + 2026-07-16 emulator-vs-real-toy testing):

- **NUM**: the monster's identity, one byte. Bytes 0x01–0x89 (1–137)
  verified accepted (with correct checksum), including byte 0x89 (num
  138), inside the presumed 12-secret-monster range; the roster is 126 regular + 12 secret
  monsters, so bytes up to ~137 are plausibly valid — the early failures
  at 0x2A/0x39 were checksum mismatches, not range errors (0x2A was later
  accepted with the right checksum).
  The number the toy *displays* is not simply byte + 1: byte 0x0A shows
  as monster "11" (= byte + 1) but byte 0x04 shows as "15". Candidate
  formula: display = (check − 1) × 10 + byte + 1, which fits 0x01→2,
  0x04→15, 0x0A→11, but predicts Diamond Back (0x11, check 2) displays
  as "28" (its earlier "#18" attribution may have been derived, not
  observed) and that no monster can display "18" — check on a real unit;
  harvest-mode dumps of known monsters will settle it.
- **check** *(fully cracked by `analysis/nibble_rule.py`)*: a checksum
  over the number byte, the BCD-encoded HP byte, AND EXP, stored in the
  4-bit field:

  ```
  digitSum(x) = hi_nibble(x) + lo_nibble(x)
  expTerm     = (EXP mod 8) − (EXP div 8)
  check = (−digitSum(NUM) − 2·digitSum(bcd(HP)) + expTerm) mod 8
  ```

  Cracked in three passes against a real toy, all on 2026-07-16. Pass 1,
  the NUM term (10 accepted payloads at HP=63/EXP=9 — nums 42, 43, 44,
  12, 135, 136, 15, 49, 123 — plus 3 rejections). Pass 2, the HP term,
  via a controlled sweep on one fixed number (138) across HP=1, 2, 3, 4,
  10 (6/6 exact — HP=10 decided between the only two linear-in-HP-digit
  models the earlier points left standing). Pass 3, the EXP term, via a
  sweep on the same fixed number+HP across EXP=1..9 and 16: the naive
  "+EXP" pattern holds for EXP=1..7 but appears to break at EXP=8 —
  until retesting found some values have a confirmed **twin 8 apart**
  (EXP=3 accepts both nibble 8 and 0; EXP=4 accepts both 9 and 1),
  revealing that **only the low 3 bits of the field are checked** — the
  top bit is a don't-care. A second EXP sweep on a different number
  (135) confirmed EXP's contribution is separable (same shift
  regardless of which number/HP is being sent).
  `expTerm(9) == 0`, which is exactly why every number and HP experiment
  in this project "just worked" while EXP sat at the emulator's default
  of 9 — its contribution was silently cancelling out the entire time.
  This formula fits every sample ever gathered in this project,
  including the two original trade.csv captures (Night Lurk, Diamond
  Back) and four HARVEST_MODE dumps that earlier looked like outliers —
  they simply weren't at EXP=9.
  Supersedes an earlier mod-4 "residue 0 wraps to 1" theory (artifact of
  too small a dataset: 0 is a plain valid check value). The nibble
  carries no monster identity — earlier "level"/"tribe"/"category"
  readings are all dead. Level, for the record, is never transmitted:
  the toy derives it from experience (one level per 30 EXP, per the
  manual), which is why received monsters with low EXP always show
  level 1.
- **HP twice, BCD, no confirmed ceiling below 99**: current HP duplicated,
  **BCD-encoded**. Verified: byte 0x63 (BCD 63) is accepted and displays
  as HP 63; 0x3E (low nibble > 9 → invalid BCD) → ERROR. **BCD 0x99 (99)
  is also accepted** (2026-07-16, with the correct HP-aware checksum,
  §3.5's `check` field) — the earlier "0x99 → ERROR" finding predates
  the discovery that HP feeds the checksum, so that rejection was almost
  certainly a checksum mismatch misread as a range/ceiling error, the
  same trap the NUM field fell into before its checksum was understood.
  The toy's own UI visibly breaks/glitches displaying HP 99 (the in-game
  balance cap is meant to be ~63), so it's accepted at the wire level
  but not a value the real game ever produces. 99 is also the natural
  ceiling of the sketch's decimal `MONSTER_HP` → `toBcd()` path (values
  ≥ 100 would emit a non-BCD high nibble, independently hitting the
  invalid-BCD ERROR) — untested whether the toy enforces any cap beyond
  BCD validity itself. The captured values (3, 1) are single-digit so
  BCD == binary there, which is why the capture alone couldn't reveal
  the encoding. Both samples identical — likely redundancy, possibly
  (current, max). Send the same value in both.
- **zeros (12 bit)**: unknown, zero in both samples. Send zeros.
- **EXP** *(checksum contribution solved, display transform still open)*:
  experience = winning-battle counter (one per win, per the manual; the
  toy computes level as EXP/30 + 1). Stored as a **raw 7-bit binary**
  value: an injected EXP=127 monster displayed "experience 15" but
  re-sent 127 bit-exact when traded back, so the storage is raw and only
  the *display* transform is unknown (low nibble? clamp at 15?). EXP now
  feeds the checksum (see §3.5's `check` field — solved for EXP 0..16,
  formula extends to the full 7-bit range); any value can be sent with
  `MONSTER_NIBBLE=0xFF`. Verified accepted: 0, 1..9, 16, 127; several
  other values made the receiver ERROR in early tests (exact values not
  recorded, and predate the checksum fix — those failures were plausibly
  checksum mismatches rather than genuine EXP-validity errors, the same
  trap NUM/HP fell into; worth re-testing now that the checksum is
  solved). Field width 7 vs 8 bits (44–51 with no stop bit) still
  ambiguous from the capture alone.
- **No checksum**: the trailing bits do not depend on NUM/HP (exhaustive
  CRC-8/CRC-16, weighted sums, nibble sums all fail; the two captures and
  all accepted emulator payloads are consistent with independent fields).

The slave's payload starts on the **cycle immediately following** the last
cycle of the master's payload. The master acks (§3.4) after the slave
payload ends.

## 4. Byte vocabulary

| Byte | Sender | Frame | Meaning (as observed) |
|------|--------|-------|-----------------------|
| 0x32 | master | §3.1 | Beacon / idle poll — "are you there / still there" |
| 0x2B | master | §3.1 | Fast poll — master is in a waiting state (selection made / accept pressed), polls at 50 Hz |
| 0x39 | master | §3.3 event | Session confirm request, sent twice ~0.5 s after link-up (devices show "Ok!") |
| 0x34 | slave | §3.2 | Ack / "I'm here, no news" |
| 0x2D | slave | §3.2 | "Ready / proceed" — advances the session to the next stage (after 0x39: session confirmed; during fast poll: triggers payload exchange; at end: trade complete, session closes) |
| 0x27 | slave | §3.3 event | Event notification — sent once, replacing 0x34, right after a slave-side state change (user action / new screen) |

Unknown: reject/cancel codes (this capture is an accepted trade on both
sides). A rejecting device presumably answers something other than
0x27/0x2D — capture one to find out.

## 5. Session flow (annotated capture timeline)

Cycle = clock cycle index in the capture; t = seconds. The complete decoded
session (every frame, produced by `analysis/full_decode.py`) is in
`analysis/transcript.txt`.

| t (s) | What happens on the wire |
|-------|--------------------------|
| 0.00–1.26 | Silence, all lines high. |
| 1.26 | **Handshake**: master starts clock bursts. Burst = ~44–100 ms of clock with idle data, then 6 × 0x32 beacons 21.3 ms apart (every 30 cycles), then clock **stops for ~21 ms** (listen gap — per prior research the device listens for a peer's beacon here; the peer that hears one first becomes slave). 4 bursts with no answer. |
| 2.29 | Burst 5: slave answers the 2nd beacon with 0x34 → **linked**. Master switches to polling 0x32 every ~100 ms (141 cycles) and never stops the clock again. Two slave replies at 2.43/2.55 are garbled/aborted mid-frame (sync stumbles); master simply re-polls ~20 ms later. |
| 2.79 | Master: 0x32→0x34, then **0x39**→0x34, then **0x39**→**0x2D**. Session confirmed — this is when both devices show "Ok!". |
| 2.9–5.3 | Idle polling 0x32/0x34 every 100 ms. Users navigate to trade / select. |
| 5.39 | Master switches to **fast poll 0x2B** every 19.9 ms (28 cycles) — master-side user action (monster selected). Slave acks 0x34. |
| 6.40 | Slave misses 2 polls (busy), then replies **0x27** once (slave-side user action), then 0x34 again. |
| 7.73 | Slave replies **0x2D** ("ready to exchange"). Master acks, waits 2 idle cycles, then sends its **52-cycle payload** (Night Lurk). Slave sends its payload (Diamond Back) starting the next cycle. Master acks. |
| 7.94 | Back to 0x32 polls @100 ms. First reply is **0x27** (slave state → showing incoming-monster preview), then 0x34s. Users look at the preview for ~3.4 s. |
| 11.36 | Master fast-polls **0x2B** again (master user pressed Accept). Slave 0x34s. |
| 12.52 | Slave **0x27** once (slave user pressed Accept), then 0x34s — with occasional missed polls (busy committing/animating). |
| 13.67 | Slave replies **0x2D** (trade complete). Master acks, parks the clock, session over. Lines return to idle high. |

Note on asymmetry: only the master's user-actions change the master's poll
byte (0x32↔0x2B); the slave signals its user-actions with 0x27/0x2D replies.
Whether roles feel symmetric to the users is invisible at this layer — the
protocol itself is strictly master-driven. What triggers each poll-mode
switch is inferred from timing against the known user flow.

## 7. Open questions

- Reject / cancel byte codes (need a capture of a refused trade).
- Meaning of the master frame's 1-cycle low trailer.
- EXP field encoding: the checksum contribution is solved (see §3.5), so
  what's left is the *display* transform — what exactly the receiver
  shows for a given raw EXP value — and the 7- vs 8-bit width. Any
  earlier "ERROR" results for specific EXP values predate the checksum
  fix and were plausibly checksum mismatches, not real EXP-validity
  errors — worth re-testing with `MONSTER_NIBBLE=0xFF` before assuming
  they're genuine. Test matrix: 16 (0x10 — shows 10 if BCD), 21 (0x15 —
  shows 15 if BCD), 64, 112.
- HP ceiling (see §3.5): BCD 99 confirmed accepted (game UI breaks, wire
  doesn't care); whether the toy enforces any cap beyond BCD validity
  itself (i.e. anything reachable only via a non-standard, non-BCD wire
  byte) is untested and would need bypassing the sketch's decimal
  `MONSTER_HP` parameter.
- Wire number ↔ displayed number mapping (see §3.5 candidate formula):
  verify what Diamond Back (byte 0x11) displays — 18 or 28 — and whether
  any monster displays "18". Harvest-mode logging of every real monster
  (name + displayed number + wire payload) resolves this empirically.
- Valid range of the number byte: 0x89 (138) confirmed (used throughout
  the HP/EXP checksum sweeps); are bytes up to 0x7D (126) and the rest of the
  12 secret monsters (presumably 0x7E..0x89, 127..138) all accepted?
- Trade-abort behavior: after an aborted exchange (slave goes silent),
  the master re-sends the *same* monster payload in the next session
  regardless of a new selection — back fully out of the toy's trade menu
  (or cancel on the toy) between harvest rounds to reset it.
- Cancel/reject codes: the emulator now logs any master byte outside
  {0x32, 0x2B, 0x39} — cancelling a trade on the toy mid-session should
  finally reveal them.
- The 12 zero bits in the payload (max HP? status flags? EXP high digits?).
- Receiver validation, updated: nibble checksum and HP BCD-ness/range are
  validated (ERROR screen + link abort on violation); a number-byte range
  check is now unconfirmed (all previously assumed range errors were
  checksum mismatches).
- Clock arbitration between two real devices (who becomes master) — this
  capture starts with device 1 already beaconing; the listen gaps are
  visible but a two-master contention was not captured.
